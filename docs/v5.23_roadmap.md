## 2. 詳細 pitfall 候選

### P1 — `cap_coverage_report()` API in `three_tier_loader`

**Problem**: v5.22 Stage B-0 是 standalone 量化腳本 `quantify_v522_tech_ma50_pitfall.py`,每次 audit 要重跑整個 N=50000 sweep。

**Fix**:
```python
# three_tier_loader.py 加 API
def cap_coverage_report(
    score_fn: Callable,
    param_name: str,
    distribution: np.ndarray,
    n_samples: int = 50000,
) -> dict:
    """Return cap-zone coverage for given param: {coverage: float, is_by_design: bool}"""
    ...
```

**驗證**: pytest guard `test_cap_coverage_report_v523` 跑 fund_score_multifactor 確認 8.19% PEG cap-zone 與 v5.22 Stage B-0 一致 (Δ < 0.01)

### P2 — `realtime_quotes.py` cap-zone audit

**Problem**: 208L 檔案 v5.22 Stage B-0 沒量化,可能有類似 beta/peg 的 floor/cap pitfall。

**Fix**: 套 Stage B-0 模式 sweep realtime_quotes 內每個 numeric param,識別 cap-zone > 0.5% 的就修。

### P3 — `fixture_cache.py` TTL 24h → 7 day stale guard

**Problem**: v5.21 設計 `DEFAULT_TTL_HOURS = 24`,但 fixture refresh cycle 是 bi-weekly,L13 Stage B-0 揭示 90-day stale fixture 不算「自動 re-fetch」。

**Fix**:
```python
# fixture_cache.py
DEFAULT_TTL_HOURS = 24        # yfinance rate-limit safe
DEFAULT_STALE_TOLERANCE_DAYS = 14  # 從 7 → 14,對齊 bi-weekly refresh
```

### P4 — `social_sentiment_provider.py` cap-zone audit

**Problem**: 326L 最大檔案,social sentiment 是 v5.19 N17 修完後 v5.22 沒複檢。

**Fix**: 套 Stage B-0 N=50000 sweep + 真實 distribution scan,識別 cap-zone > 0.5% pitfall。

### P5 — `live_score_engine` cap-zone warning API

**Problem**: `live_score_engine.recompute_all_scores()` 不報 cap-zone warning,使用者不知道哪個 ticker score 是因為 cap 而不是真實分布。

**Fix**:
```python
# live_score_engine.py
def recompute_all_scores_with_cap_warnings(...) -> dict:
    """Return scores + per-ticker cap-zone warnings"""
    return {
        "scores": {...},
        "cap_warnings": {
            "3690.HK": {"peg": "PEG=28.72 → peg_factor=0.0 (clipped from decay)"},
            ...
        }
    }
```
